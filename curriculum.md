## Vibe Coding Class: 12 Classes + 1 Demo Day × 105 Minutes | Grade 6+ (Middle School Majority)

---

### Session Structure Template

Every session follows this internal rhythm. Times are targets, not rigid blocks — instructor slides between them based on room energy.

| Segment | Duration | Purpose |
|---------|----------|---------|
| Hook | 10 min | Anchor attention; show a real-world example of what today's skill produces |
| Direct instruction + live demo | 20 min | Instructor builds in real-time, narrating every prompt decision |
| Guided build | 40 min | Students build with instructor circulating; structured checkpoints at 15 and 30 min |
| Peer check | 15 min | Show your screen to one partner; partner names one thing that works and one thing that doesn't |
| Wrap + document | 20 min | Students write 2–3 sentences in their prompt log; instructor names the skill used and previews next session |

The 105-minute length is long enough for middle schoolers to enter a flow state but only if the first visible output arrives within 30 minutes. Every session is designed to produce something deployable or demonstrable before the midpoint.

---

### Phase Allocation Across 12 Classes + Demo Day

| Phase | Classes | Focus |
|-------|----------|-------|
| Orientation and First Wins | 1–2 | Tools, first output, what prompting is |
| Prompt Skills and Iteration | 3–5 | Anatomy, iteration, error recognition |
| Domain Builds | 6–10 | Independent product development |
| Capstone Prep | 11–12 | Polish, documentation, rehearsal |
| Demo Day | separate day, after Class 12 | Live presentations |

---

### Session-by-Session Plan

---

**Session 1 — What Is This? (Orientation)**

*Theme: Lower the barrier. Ship something in the first class.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor shows a working app (quiz, game, or tool) built entirely through prompts. No code shown. "We built this without writing a single line of code." |
| 0:10–0:30 | Live demo: Instructor builds a simple "about me" webpage on Lovable using only typed prompts. Narrates every decision: why this wording, what changed, what went wrong. |
| 0:30–1:10 | Students replicate: each student builds their own "about me" page. One starting prompt. One revision. Goal is a live URL before the end of this block. |
| 1:10–1:25 | Partner share: open each other's URL. Does it match what you described? Write one mismatch in the prompt log. |
| 1:25–1:45 | Wrap: Instructor explains the term vibe coding, its origin (Karpathy), and what this class will produce by week 12. Students write one sentence: "I want to build ___." |

*Deliverable by end of session: Deployed personal page, live URL.*

---

**Session 2 — Scratch Bridge and Logic Concepts**

*Theme: What does code actually do? Build the mental model before the abstraction.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Show the same animation built two ways — block-based (Scratch) and prompt-based (Replit). Ask: what's the same? What's different? |
| 0:10–0:25 | Direct instruction: sequence, events, inputs, outputs — the four concepts that underlie everything. Use Scratch to make them visible, not because Scratch is the point. |
| 0:25–1:05 | Students build one simple Scratch animation (sprite moves, responds to a key press, changes color). Then open Replit and prompt AI to produce the same behavior in a webpage. Focus: same logic, different surface. |
| 1:05–1:20 | Peer check: partner tests the Replit version. Does clicking or pressing a key do something? If not, student writes the re-prompt that should fix it — no manual editing. |
| 1:20–1:45 | Wrap: Side-by-side comparison on screen. Instructor explains: when you vibe code, you are still writing logic — you are writing it in English, not Python. Prompt log: one thing the AI understood, one thing it didn't. |

*Deliverable by end of session: Simple interactive Replit page; annotated prompt log entry.*

---

**Session 3 — Prompt Anatomy**

*Theme: A vague prompt produces a vague result. Precision is a skill.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor runs three versions of the same vague prompt live. All three produce different, wrong results. "Why did this happen?" |
| 0:10–0:30 | Direct instruction: four-part prompt structure — Role, Task, Format, Constraints. Walk through one well-formed example for each part. Students copy the framework into their prompt log as a reference. |
| 0:30–1:10 | Activity: Students receive a target app description (a simple to-do list). They write three prompts — no structure, partial structure, full structure — and compare outputs. They annotate which output was closest to the target and why. |
| 1:10–1:25 | Peer check: swap prompt logs. Partner rates each prompt on how precisely it used the four-part structure. |
| 1:25–1:45 | Wrap: Class votes on the best prompt submitted. Instructor deconstructs it. Students revise their weakest prompt in the log. |

*Deliverable by end of session: Three annotated prompts showing progression; working to-do list app.*

---

**Session 4 — When AI Is Wrong**

*Theme: AI will always produce something. That does not mean it produced the right thing.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor opens a deliberately broken app. Asks: "This is what the AI made. What's wrong?" Students call out errors verbally. |
| 0:10–0:25 | Direct instruction: three types of AI output problems — it misunderstood the task, it technically worked but is wrong for the use case, it worked but is unusable. How to identify each. |
| 0:25–1:05 | Students receive a pre-built app (quiz or survey) with three planted errors — one of each type. Task: identify each error, write the re-prompt that fixes it. No manual code editing allowed under any circumstances. |
| 1:05–1:20 | Partner check: swap and test the fixed app. Can the partner use it without instructions? |
| 1:20–1:45 | Wrap: Instructor debriefs each error type. Key message: the job of the vibe coder is not to write code — it is to evaluate output and re-prompt with precision. Prompt log: document all three fixes. |

*Deliverable by end of session: Fixed app + minimum 3-entry prompt log showing error → re-prompt → result.*

---

**Session 5 — Forms, Data, and the Ethics Hook**

*Theme: When your app collects information, something happens to it. You are responsible for deciding what.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor asks students to fill in a fictional form on screen: name, age, school, favorite class. Then: "Where does this go? Who can see it?" |
| 0:10–0:25 | Direct instruction: how forms work (input → storage → display). What data privacy means at the simplest level. Tied directly to what students will build, not to abstract ethics. |
| 0:25–1:05 | Students build a quiz or survey app with a form. Must include at least two fields and display the result on screen. Then: write a one-sentence privacy note that tells users what their data is used for and prompt AI to add it to the app's UI. |
| 1:05–1:20 | Peer check: partner fills in the form and reads the privacy note. Is the note accurate? Is the form usable? |
| 1:20–1:45 | Wrap: Instructor names the pattern: every product in weeks 6–9 that collects user data needs a privacy decision. This is not optional. Prompt log: document the form app and the privacy note prompt. |

*Deliverable by end of session: Working quiz/survey app with deployed privacy note.*

---

**Session 6 — Track Selection and Scoping**

*Theme: You cannot build what you cannot describe. Scope before you build.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Show three student-grade products from prior cohorts (or equivalent examples). "All of these were built in four sessions. Which would you use?" |
| 0:10–0:30 | Track overview. Instructor walks through all four tracks with examples at both grade 6 and grade 9 ceiling: School Life, Community, Creative, Games. |
| 0:30–0:55 | Students write their product brief: who uses it, what problem it solves, what the simplest working version looks like. Grade 6–7: one paragraph. Grade 8–10: one structured page. Instructor circulates and gates scope — overbuild is the primary failure mode at this stage. |
| 0:55–1:20 | Students begin first prototype: one-prompt skeleton. The goal is a working shell, not a finished product. Navigation works. At least one screen displays. No polish. |
| 1:20–1:45 | Partner share: read each other's brief, then open the prototype. Does what's on screen match what the brief says? Verbal feedback only. Prompt log: brief pasted in, first prompt documented. |

*Deliverable by end of session: Approved product brief; working one-screen prototype.*

---

**Session 7 — Build Sprint 1**

*Theme: Get it working before you make it good.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor defines "working" vs. "finished." A working product can be used by someone who has never seen it before. A finished product is also polished. Today's goal is working, not finished. |
| 0:10–0:20 | Students read their brief aloud to themselves, identify the one core feature that makes the product function, and write the prompt for it before touching the tool. |
| 0:20–1:10 | Build sprint. Instructor circulates. At the 30-minute mark, every student must have something functional on screen — if not, instructor intervenes on prompt formulation, not code. At 50 minutes, students must test their product by using it themselves as if they are the target user. |
| 1:10–1:25 | Peer usability test: partner uses the product without being told how. Student observes silently. Notes what breaks or confuses without explaining or helping. |
| 1:25–1:45 | Wrap: Prompt log documents the main build prompts and the first usability finding. Students write one re-prompt they plan to write next session based on what the test revealed. |

*Deliverable by end of session: Core feature functional; first usability finding documented.*

---

**Session 8 — Build Sprint 2 + Feature Add**

*Theme: One feature at a time. Scope creep kills student projects.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor shows a student project (anonymized) that added five features at once and broke. Contrast with one that added one feature, tested it, and moved on. |
| 0:10–0:20 | Students identify the one feature they will add today based on last session's usability finding. Write the prompt before touching the tool. |
| 0:20–1:05 | Build sprint. Instructor enforces the one-feature rule actively. Grade 9–10 students begin connecting an external API or adding a second user state. |
| 1:05–1:25 | Second usability test with a different partner. Same protocol: partner uses it silently, student observes. |
| 1:25–1:45 | Prompt log: document feature add prompt, what happened, what the second test revealed. Write one revision prompt for next session. |

*Grade differentiation: Grade 6–7 product has two working features. Grade 8 has two features plus navigation. Grade 9–10 has external data integration or multi-state logic.*

*Deliverable by end of session: Product with second feature; two usability tests documented.*

---

**Session 9 — Build Sprint 3 + Feature Refinement**

*Theme: Depth over breadth. Make what exists work well before adding more.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor shows two versions of the same feature — one added quickly and left rough, one refined after a second pass. Ask: which would you rather demo? |
| 0:10–0:20 | Students review both rounds of usability findings (sessions 7–8) and pick the single highest-impact fix or addition. Write the prompt before touching the tool. |
| 0:20–1:05 | Build sprint. Instructor enforces the one-change-at-a-time rule. Grade 9–10 students who already have external data or multi-state logic working now focus on edge cases: empty input, wrong input, no results. |
| 1:05–1:25 | Third usability test with a new partner. Same protocol: partner uses it silently, student observes and records without helping. |
| 1:25–1:45 | Prompt log: document the refinement prompt, what changed, and the third usability finding. Students flag anything they still consider broken heading into the ethical review next session. |

*Grade differentiation: Grade 6–7 refines one feature for reliability. Grade 8 refines one feature and adds a basic error state. Grade 9–10 handles at least two edge cases in their external data or multi-state logic.*

*Deliverable by end of session: Refined product; third usability test documented.*

---

**Session 10 — Revision and Ethical Review**

*Theme: Does this product behave the way you intended? Does it behave fairly?*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor runs a brief on one real-world AI product that worked as designed but caused harm (keep age-appropriate: autocomplete bias, recommendation loops). "Did the builder intend this?" |
| 0:10–0:25 | Students apply a three-question ethical check to their own product: Does it collect any user data? Could it exclude or misrepresent any group of users? If it broke or gave wrong output, who would be affected? Write answers in the prompt log. |
| 0:25–1:05 | Revision sprint based on the ethical check and usability findings from sessions 7–9. At least one change must address a usability problem. At least one must address the ethical check. |
| 1:05–1:25 | Cross-group test: students in different tracks test each other's products. Written feedback card: one thing that works, one thing that confuses, one question the product raises. |
| 1:25–1:45 | Wrap: Prompt log finalized for domain build phase. Product brief updated to reflect what was actually built vs. what was planned. |

*Deliverable by end of session: Revised product; completed ethical check; updated brief.*

---

**Session 11 — Capstone Prep: Documentation and Prompt Log**

*Theme: The product is the evidence. The documentation is the argument.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor explains what Demo Day looks like — live URL, 5-minute presentation, two required answers: what did you build and what did AI get wrong? |
| 0:10–0:25 | Capstone requirements walkthrough. Rubric distributed and read aloud. Students self-assess current product against rubric and identify gaps. |
| 0:25–1:05 | Documentation sprint. Students compile prompt log (minimum 8 annotated entries), finalize product brief, and write the "what AI got wrong" section — at least two examples with re-prompts shown. Instructor circulates, reads drafts, flags missing entries. |
| 1:05–1:25 | Peer review of documentation. Partner checks: is every prompt log entry annotated with what changed and why? Is the brief readable by someone who has never used the product? |
| 1:25–1:45 | Final build time: students address any remaining functional gaps flagged during documentation review. |

*Deliverable by end of session: Prompt log assembled; product brief final draft; product at submission-ready state.*

---

**Session 12 — Rehearsal Demos**

*Theme: Presenting a product is a different skill from building one.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Hook: Instructor models a 5-minute demo — including one moment where the product doesn't work as expected and the presenter handles it without panic. |
| 0:10–0:20 | Students write a talk track outline: problem statement (30 sec), product demo (2.5 min), what AI got wrong (1 min), what they would add next (30 sec). |
| 0:20–1:10 | Small group rehearsals: groups of 3–4. Each student presents full 5 minutes. Group completes a feedback card per presenter: Was the problem clear? Did the demo work? Did they explain the AI failure convincingly? |
| 1:10–1:35 | Students revise their talk track based on feedback. Any final product fixes are made. |
| 1:35–1:45 | Wrap: final logistics for Demo Day. URL submitted to instructor. All documentation uploaded or handed in. |

*Deliverable by end of session: Talk track finalized; all capstone documents submitted; URL confirmed working.*

---

**Demo Day — Ship It**

*Held as a separate day after the 12 regular classes conclude — not one of the 12 sessions itself. Same 105-minute format; scheduled roughly a week after Class 12 so URLs stay fresh and instructors have time to confirm logistics.*

| Time | Activity |
|------|----------|
| 0:00–0:10 | Setup: students open their products, test URLs, silence notifications. Instructor randomizes presentation order. |
| 0:10–1:25 | Live demos. Each student/team: 5 minutes presentation + 1 minute Q&A from peers. Audience uses a structured observation card: one thing this product does well, one question they have, one feature they would add. |
| 1:25–1:40 | Class vote (non-graded) across four categories: Most Useful, Most Creative, Best Iteration Story, Best Ethical Reasoning. |
| 1:40–1:45 | Instructor closes with one statement per student — specific, not generic. |

*No written work due on Demo Day. All documentation was submitted in Class 11.*

---

### Capstone Deliverables (Unchanged in Content, Adjusted for 12-Class + Demo Day Timeline)

| Deliverable | Grade 6–7 format | Grade 8–10 format |
|-------------|-----------------|-------------------|
| Deployed product | Live URL, core feature functional | Live URL, multi-feature, edge case handled |
| Prompt log | 8+ entries, informal annotation | 8+ entries, structured table with what/why/result columns |
| Product brief | One paragraph | One page: problem, features, limitations, ethical check |
| Demo | 4–5 min | 5–6 min including technical reflection |
| Additional (Gr 9–10 only) | — | Written answer: what would production-readiness require? |

|-----------|-----------|-----------|
| Working product | 45% | 35% |
| Prompt log (iteration quality) | 25% | 30% |
| Demo and communication | 20% | 20% |
| Brief and reflection | 10% | 15% |
